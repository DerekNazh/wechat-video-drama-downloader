# 功能与异常情况发生与数据库一致性的关系

本文档记录各功能操作和异常情况对数据库一致性的影响，帮助开发者理解数据流向和处理策略。

---

## 实体生命周期

### 实体关系图

```
┌─────────────┐       ┌──────────────┐       ┌────────────────┐
│   Author    │ 1───N │ AuthorVideo  │ 1───1 │ DownloadTask   │
│  (作者)     │       │  (视频)       │       │  (下载任务)    │
└─────────────┘       └──────────────┘       └────────────────┘
     │                      │                       │
     │                      │                       │
     ▼                      ▼                       ▼
 app.db.authors      app.db.author_videos    app.db.download_tasks
 - id                - video_id (PK)          - task_id (PK)
 - name              - author_id (FK)         - video_id (FK)
 - source_author_id  - title                  - status
 - latest_publish... - is_downloaded          - progress
                     - download_path          - completed_at
```

### 生命周期概览

| 实体 | 创建 | 更新 | 删除 | 关联数据库 |
|------|------|------|------|------------|
| Author | 添加作者 | 增量同步、修改标签 | 删除作者 | app.db |
| AuthorVideo | 添加作者时批量创建 | 标记已下载、重置状态 | 删除作者时级联删除 | app.db |
| DownloadTask | 创建下载任务 | 进度更新、状态变更 | 任务完成/取消/删除 | app.db + gopeed.db |

## 功能操作与数据库写入

### 1. 添加作者

```
用户操作：添加作者
    │
    ▼
前端请求：POST /api/search/add-author
    │
    ▼
后端处理：search.add_author_with_videos()
    │
    ├─► [1] 调用 Go 后端搜索作者
    │       client.search_authors_by_keyword()
    │       → Go 返回作者信息（不涉及数据库）
    │
    ├─► [2] 写入 app.db
    │       db.create_author(author)
    │       → authors 表新增记录
    │
    └─► [3] 获取视频列表并写入 app.db
            client.get_author_videos()
            db.create_author_video(video)
            → author_videos 表新增记录
```

**数据库影响**：
- `app.db.authors`：新增 1 条记录
- `app.db.author_videos`：新增 N 条记录
- `gopeed.db`：无影响

**一致性风险**：低（仅涉及 app.db）

**执行流程分析**：
| 阶段 | app.db 状态 | 异常影响 |
|------|-------------|----------|
| 步骤 2 前 | 无作者记录 | 用户需重新添加 |
| 步骤 2 后，步骤 3 前 | 有作者，无视频 | 作者显示 0 个视频，可增量同步 |
| 步骤 3 中断 | 有作者，部分视频 | 数据不完整，可增量同步补全 |

### 2. 创建下载任务

```
用户操作：点击下载视频
    │
    ▼
前端请求：POST /api/task/create
    │
    ▼
后端处理：TaskService.create_download_task()
    │
    ├─► [1] 查询 app.db 获取视频信息
    │       db.get_author_video(video_id)
    │
    ├─► [2] 调用 Go 后端创建任务（先写 Go）
    │       client.download_video()
    │       → Go 写入 gopeed.db.tasks
    │       → 返回 task_id
    │
    └─► [3] 写入 app.db（后写 App）
            db.create_download_task(task)
            → download_tasks 表新增记录
            → 关联 video_id
```

**数据库影响**：
- `gopeed.db.tasks`：新增 1 条记录（Go 后端写入）
- `app.db.download_tasks`：新增 1 条记录（镜像）

**一致性风险**：高（双写时序问题）

**执行流程分析**：
| 阶段 | gopeed.db 状态 | app.db 状态 | 异常影响 |
|------|----------------|-------------|----------|
| 步骤 2 前 | 无任务 | 无任务 | 无影响，可重试 |
| 步骤 2 后，步骤 3 前 | 有任务 | 无任务 | **孤立任务**：Go 有任务但 App 无法追踪 |
| 步骤 3 完成 | 有任务 | 有任务 | 正常状态 |

**孤立任务后果**：
- 监控器无法追踪该任务进度
- 并发窗口被占用（max_concurrent）
- 用户看不到下载进度

**恢复策略**：启动时 `cleanup_stale_tasks()` 清理孤立任务

### 3. 查询下载进度

```
用户操作：查看下载列表
    │
    ▼
前端请求：GET /api/task/list
    │
    ▼
后端处理：TaskService.get_downloading_tasks()
    │
    ├─► [1] 查询 app.db 活跃任务
    │       db.list_download_tasks(status=pending/running)
    │
    └─► [2] 逐个同步 Go 后端状态
            get_task_progress(task_id)
                │
                ├─► 调用 Go API 获取最新状态
                │       GET /api/task/list
                │
                └─► 写回 app.db
                        db.update_download_task_status()
                        db.update_download_task_progress()
```

**数据库影响**：
- `app.db.download_tasks`：更新 status、progress、speed 等字段

**一致性风险**：中（依赖 Go 后端响应）

**执行流程分析**：
| Go 后端状态 | app.db 状态 | 处理方式 |
|-------------|-------------|----------|
| 任务存在，状态一致 | 同步更新 | 正常 |
| 任务存在，状态不同 | 更新为 Go 状态 | 同步 |
| 任务不存在（Go 已删除） | 活跃任务记录 | 删除 app.db 记录 |
| Go 无响应 | 保持原状态 | 等待重试 |

### 4. WebSocket 进度推送

```
Go 后端事件：下载进度变化
    │
    ▼
WebSocket 推送：ws://127.0.0.1:2022/ws/downloader
    │
    ▼
App 处理：DownloadProgressListener._on_message()
    │
    ├─► [1] 解析任务状态
    │       task = json.loads(message)["data"]["task"]
    │
    └─► [2] 终态处理（done/completed）
            if status in ("done", "completed"):
                │
                ├─► 验证文件存在
                │       os.path.isfile(full_path)
                │
                ├─► 写入 app.db
                │       db.update_video_downloaded(video_id, path)
                │       db.update_download_task_status(task_id, "done")
                │
                └─► 触发事件总线
                        emit_task_completed({video_id, author_id})
```

**数据库影响**：
- `app.db.author_videos`：更新 is_downloaded=1, download_path
- `app.db.download_tasks`：更新 status=done, completed_at

**一致性风险**：高（依赖文件验证）

**执行流程分析**：
| 场景 | Go 状态 | 文件状态 | app.db 更新 | 后果 |
|------|---------|----------|-------------|------|
| 正常完成 | done | 存在 | is_downloaded=1 | 正常 |
| 文件丢失 | done | 不存在 | status=failed | 可重试 |
| 验证前崩溃 | done | 未知 | 无更新 | 启动时校验恢复 |

### 5. 删除任务

```
用户操作：取消下载
    │
    ▼
前端请求：DELETE /api/task/{task_id}
    │
    ▼
后端处理：TaskService.delete_task()
    │
    ├─► [1] 取消 Go 后端任务
    │       client.cancel_task(task_id)
    │       → Go 删除 gopeed.db 记录
    │
    ├─► [2] 删除 app.db 记录
    │       db.delete_download_task(task_id)
    │
    └─► [3] 清理本地文件
            _cleanup_local_file(task)
            → 删除视频文件
            → 重置 video.is_downloaded=0
```

**数据库影响**：
- `gopeed.db.tasks`：删除 1 条记录
- `app.db.download_tasks`：删除 1 条记录
- `app.db.author_videos`：更新 is_downloaded=0

**一致性风险**：中（双删时序问题）

**执行流程分析**：
| 阶段 | gopeed.db 状态 | app.db 状态 | 异常影响 |
|------|----------------|-------------|----------|
| 步骤 1 前 | 有任务 | 有任务 | 正常状态 |
| 步骤 1 后，步骤 2 前 | 无任务 | 有任务 | 孤立记录，查询时自动清理 |
| 步骤 2 后 | 无任务 | 无任务 | 正常状态 |

### 6. 删除作者

```
用户操作：删除作者
    │
    ▼
前端请求：DELETE /api/author/{author_id}
    │
    ▼
后端处理：author.delete_author()
    │
    ├─► [1] 查询该作者所有活跃任务
    │       db.list_download_tasks_by_author(author_id)
    │
    ├─► [2] 取消所有活跃任务
    │       for task in tasks:
    │           TaskService.delete_task(task.task_id)
    │
    └─► [3] 删除作者记录
            db.delete_author(author_id)
            → CASCADE 删除 author_videos
            → CASCADE 删除 download_tasks
```

**数据库影响**：
- `app.db.authors`：删除 1 条记录
- `app.db.author_videos`：删除 N 条记录（CASCADE）
- `app.db.download_tasks`：删除 N 条记录（CASCADE）
- `gopeed.db.tasks`：删除 N 条记录（通过 API）

**一致性风险**：中（批量操作）

**执行流程分析**：
| 阶段 | gopeed.db 状态 | app.db 状态 | 异常影响 |
|------|----------------|-------------|----------|
| 步骤 2 中断 | 部分删除 | 未删除 | 孤立任务，启动时清理 |
| 步骤 3 前 | 全部删除 | 有作者记录 | 可重新删除 |
| 步骤 3 完成 | 全部删除 | 全部删除 | 正常状态 |

## 异常情况与数据不一致

### 异常 1：应用崩溃

```
场景：App 进程异常退出
    │
    ▼
影响：
    ├─► app.db 可能未完成写入（事务回滚）
    ├─► gopeed.db 可能已完成写入
    └─► 两边状态不一致
```

**典型场景**：

| 崩溃时机 | gopeed.db 状态 | app.db 状态 | 后果 | 恢复方式 |
|----------|----------------|-------------|------|----------|
| 创建任务后（Go 写完，App 未写） | 有任务 | 无任务 | 孤立任务 | 启动时清理 |
| 完成推送后（文件验证前） | done | 无更新 | 视频未标记已下载 | 启动时校验 |
| 删除任务中途 | 可能删除 | 可能未删除 | 孤立记录 | 启动时清理 |

---

### 异常 2：Go 后端突然断开服务（关键场景）

```
场景：Go 进程异常退出（崩溃、被杀、端口冲突等）
    │
    ▼
影响：
    ├─► gopeed.db 任务丢失（进程内存数据未持久化）
    ├─► WebSocket 连接断开
    ├─► HTTP API 无响应
    └─► App 无法获取进度、无法创建新任务
```

**这是最常见且影响最大的异常场景**。Go 后端断开后，App 仍在运行，但所有下载相关功能失效。

#### 断开时机与数据状态分析

| 断开时机 | gopeed.db 状态 | app.db 状态 | 用户可见后果 | 恢复方式 |
|----------|----------------|-------------|--------------|----------|
| **任务创建中**（Go 收到请求但未写入 DB） | 无任务 | 无任务 | 前端报错"创建失败" | 用户重试 |
| **任务创建后**（Go 写入 DB，App 写入前） | 有任务 | 无任务 | **孤立任务**：Go 有任务但 App 无法追踪 | 启动时清理 |
| **下载进行中**（进度 50%） | 任务丢失 | progress=50% | 进度卡住，显示"正在下载"但实际已停止 | HTTP 查询发现任务不存在 → 删除记录 |
| **下载即将完成**（99%） | 任务丢失 | progress=99% | 显示"99%"但实际未完成，文件可能不完整 | 启动时校验文件完整性 |
| **下载刚完成**（Go 报 done，WebSocket 未推送） | done（丢失） | running | 视频已下载但未标记，下次启动校验恢复 | 启动时校验 |
| **下载刚完成**（Go 报 done，WebSocket 推送中） | done（丢失） | done | 视频已标记但文件可能不完整 | 启动时校验文件 |

#### 断开后的连锁反应

```
Go 后端断开
    │
    ├─► [1] WebSocket 连接断开
    │       socket_client._on_close() 触发
    │       → 3 秒后自动重连
    │       → 重连失败（Go 未恢复）
    │       → 循环重连，日志警告
    │
    ├─► [2] HTTP API 无响应
    │       所有 client.*() 调用超时（30s）
    │       → 创建任务失败
    │       → 查询进度失败
    │       → 搜索作者失败
    │
    ├─► [3] 监控器阻塞
    │       _monitor_loop() 中：
    │       → 同步视频失败（client.get_author_videos 超时）
    │       → 创建任务失败（client.download_video 超时）
    │       → 等待任务完成（get_downloading_tasks 返回旧数据）
    │       → 监控器卡在等待循环
    │
    ├─► [4] 前端状态混乱
    │       SSE 推送 service_online=false
    │       → 前端显示"服务离线"
    │       → 但下载列表仍显示旧进度
    │       → 用户无法操作
    │
    └─► [5] 数据不一致累积
            app.db 中：
            → 活跃任务记录残留（running/pending）
            → 进度数据过时
            → 视频未标记已下载（实际已下载）
```

#### 关键问题：监控器并发窗口被占用

```
监控器滑动窗口逻辑：
    while active_count >= max_concurrent:
        wait 5 seconds

Go 断开后：
    get_downloading_tasks() 返回旧数据（无法同步 Go）
    → active_count 仍为 max_concurrent
    → 监控器卡在等待循环
    → 无法创建新任务
    → 用户看不到任何下载进度
```

**后果**：监控器完全阻塞，用户无感知，只能等待 App 重启。

#### 当前处理策略

| 层面 | 现有策略 | 效果 |
|------|----------|------|
| **WebSocket** | 自动重连（3s 间隔） | Go 恢复后自动恢复 |
| **HTTP API** | 超时后返回错误 | 前端报错，用户可重试 |
| **监控器** | 无特殊处理 | 卡在等待循环，需重启 |
| **前端状态** | SSE 推送 service_online=false | 显示"服务离线" |
| **启动恢复** | 清空 gopeed.db + 清理残留任务 | 完全恢复一致 |

#### 缺失的处理（待改进）

| 问题 | 当前状态 | 建议改进 |
|------|----------|----------|
| 监控器阻塞 | 无处理，卡在等待 | 检测 Go 离线后暂停监控，恢复后自动重启 |
| 进度查询超时 | 返回旧数据 | 检测 Go 离线后标记任务为 "disconnected" |
| 孤立任务清理 | 仅启动时清理 | 运行时定期清理（如每 5 分钟） |
| Go 自动重启 | 仅启动时启动 | 检测 Go 离线后尝试重启 |

### 异常 6：下载中断导致损坏文件残留（关键场景）

```
场景：Go 后端下载过程中断（崩溃、网络中断、App 关闭）
    │
    ▼
影响：
    ├─► 磁盘留下不完整的 .mp4 文件
    ├─► 文件名占用最终文件名（非临时文件）
    └─► 重新下载时文件名冲突
```

#### Go 后端下载机制分析

```
Go 后端下载流程：
    │
    ├─► [1] 确定文件名
    │       name := f.meta.SingleFilepath()
    │       → 例如: "2026-05-20_视频标题_1080p.mp4"
    │
    ├─► [2] 创建/打开文件
    │       if 文件不存在:
    │           f.file = os.Create(name)  ← 直接创建最终文件
    │       else:
    │           f.file = os.OpenFile(name)  ← 断点续传
    │
    └─► [3] 写入数据
            f.file.WriteAt(buf, offset)
            → 直接写入最终文件，无临时文件
```

**关键发现**：Go 后端**不使用临时文件**，直接写入最终 `.mp4` 文件。

#### 损坏文件残留场景

| 中断时机 | 磁盘文件状态 | app.db 状态 | 后果 |
|----------|--------------|-------------|------|
| **下载 0%**（刚创建文件） | 0 字节 .mp4 文件 | 无任务记录 | 空文件，可被覆盖 |
| **下载 50%**（写入一半） | 部分数据 .mp4 文件 | progress=50% | 损坏文件，无法播放 |
| **下载 99%**（即将完成） | 几乎完整 .mp4 文件 | progress=99% | 可能可播放，但不完整 |
| **下载 100%**（未关闭文件） | 完整数据，未 flush | progress=100% | 可能可播放 |

#### 文件名冲突问题

```
文件命名规则：
    filename = "{date}_{title}_{spec}.mp4"
    例如: "2026-05-20_测试视频_1080p.mp4"

问题：
    1. 文件名基于视频标题，不是 video_id
    2. 同一视频多次下载，文件名相同
    3. 损坏文件占用文件名，新下载会覆盖还是追加？
```

**Go 后端行为**：
- 文件存在 → 打开并追加（断点续传）
- 损坏文件 → 可能导致新下载数据追加到损坏数据后面 → 文件损坏

#### 损坏文件检测难点

```
检测损坏文件的挑战：
    │
    ├─► [1] 文件大小不可靠
    │       损坏文件可能接近预期大小
    │       无法通过大小判断完整性
    │
    ├─► [2] 文件格式检测复杂
    │       MP4 文件需要解析 moov atom
    │       简单的文件存在性检查无法发现损坏
    │
    └─► [3] 数据库状态不可靠
            app.db 中 progress < 100% 但文件存在
            无法区分"下载中"和"下载中断"
```

#### 当前处理策略

| 场景 | 现有处理 | 效果 |
|------|----------|------|
| **停止监控** | 删除活跃任务的视频文件 | 清理损坏文件 |
| **启动时校验** | 检查 is_downloaded=1 的文件是否存在 | 不处理未完成的下载 |
| **重新下载** | Go 后端断点续传 | 可能追加到损坏文件 |

#### 缺失的处理

| 问题 | 当前状态 | 建议改进 |
|------|----------|----------|
| 损坏文件检测 | 无 | 检查 progress < 100% 且文件存在的记录 |
| 损坏文件清理 | 仅停止监控时 | 启动时清理未完成下载的文件 |
| 文件名冲突 | Go 后端追加写入 | 删除损坏文件后重新下载 |
| 下载前检查 | 无 | 创建任务前检查是否有损坏文件 |

#### 损坏文件残留的连锁反应

```
损坏文件残留
    │
    ├─► [1] 重新下载时断点续传
    │       Go 后端打开损坏文件
    │       → 追加新数据到损坏数据后面
    │       → 文件仍然损坏
    │
    ├─► [2] 监控器认为任务完成
    │       Go 报 done（文件写入完成）
    │       → App 验证文件存在
    │       → 文件存在但损坏
    │       → 标记 is_downloaded=1
    │       → 用户播放失败
    │
    └─► [3] 用户手动删除后状态不一致
            用户删除损坏文件
            → app.db 仍标记 is_downloaded=1
            → 监控器不会重新下载
            → 启动时校验可修复
```

    │       db.list_author_videos() → 筛选 is_downloaded=0
    │       → 按时间排序
    │
    ├─► [3] 滑动窗口创建任务
    │       while task_index < len(all_undownloaded):
    │           │
    │           ├─► 查询活跃任务数
    │           │       task_service.get_downloading_tasks()
    │           │       → 调用 Go API 同步状态
    │           │       → 返回活跃任务列表
    │           │

### Go 后端断开时的影响

```
Go 后端断开（进程崩溃、端口冲突、手动停止）
    │
    ▼
[阶段 1] 同步视频失败
    │
    │   video_service.add_all_authors_latest_videos()
    │   → client.get_author_videos() 超时（30s）
    │   → 异常捕获，日志记录
    │   → 本轮同步跳过，进入下一轮
    │
    ▼
[阶段 2] 收集未下载视频（不受影响）
    │
    │   db.list_author_videos() → 纯 app.db 操作
    │   → 正常返回未下载列表
    │
    ▼
[阶段 3] 滑动窗口创建任务（阻塞）
    │
    │   task_service.get_downloading_tasks()
    │   → 调用 Go API 同步状态 → 超时（30s）
    │   → 返回旧数据（无法同步）
    │   → active_count = max_concurrent（旧数据）
    │
    │   if active_count >= max_concurrent:
    │       wait 5 seconds
    │       continue  ← 卡在这里！
    │
    │   每次循环：
    │   - 查询活跃任务 → 超时 → 返回旧数据
    │   - 窗口满 → 等待 5 秒
    │   - 无限循环，无法创建新任务
    │
    ▼
[阶段 4] 监控器完全阻塞
    │
    │   用户可见：
    │   - 前端显示"服务离线"
    │   - 下载列表显示旧进度（卡住）
    │   - 无法添加新作者（搜索超时）
    │   - 无法手动下载（创建任务超时）
    │
    │   后台状态：
    │   - 监控线程卡在等待循环
    │   - WebSocket 线程循环重连
    │   - SSE 状态推送线程检测到离线
    │
    ▼
[恢复] 用户重启 App
    │
    │   启动时：
    │   - 清空 gopeed.db
    │   - 清理 app.db 残留任务
    │   - 校验文件完整性
    │   - 重启 Go 后端
    │   - 监控器重新启动
    │
    ▼
应用恢复正常
```

### 关键代码路径分析

| 代码位置 | Go 断开时的行为 | 影响 |
|----------|-----------------|------|
| `core/service/task.py:get_downloading_tasks()` | Go API 超时 → 返回 app.db 旧数据 | 进度过时 |
| `core/service/task.py:get_task_progress()` | Go API 超时 → 返回 None | 任务状态未知 |
| `core/service/task.py:create_download_task()` | Go API 超时 → 返回 `{"code": -1}` | 无法创建任务 |
| `core/monitor/monitor.py:_monitor_loop()` | 等待循环卡住 | 监控器阻塞 |
| `core/utils/socket_client.py:_run()` | WebSocket 断开 → 3s 重连 | 循环重连 |
| `core/utils/service_status_push.py:_status_loop()` | 检测到离线 → SSE 推送 | 前端显示离线 |

### 数据一致性风险总结

| 风险点 | 描述 | 恢复方式 |
|--------|------|----------|
| **进度数据过时** | app.db 中 progress/speed 未更新 | 启动时清理残留任务 |
| **孤立任务累积** | Go 有任务但 App 无记录（创建中断） | 启动时清理 |
| **僵尸任务残留** | App 有活跃任务但 Go 无（Go 断开） | 启动时清理 |
| **文件状态不一致** | 视频已下载但未标记（WebSocket 中断） | 启动时校验文件 |
| **监控器阻塞** | 滑动窗口卡在等待循环 | 用户重启 App |

## 一致性保证总结

| 层面 | 策略 | 效果 |
|------|------|------|
| **启动时** | 清空 Go 数据库 + 清理残留任务 + 校验文件 | 恢复到一致状态 |
| **运行时** | WebSocket 推送 + HTTP 轮询兜底 | 实时同步进度 |
| **终态时** | 验证文件存在 | 防止误报完成 |
| **异常时** | 自动重连 + 启动恢复 | 最终一致 |

**核心原则**：
1. **不信任 Go 后端的 done 状态**：必须验证文件存在
2. **App 数据库为主**：Go 数据库视为临时缓存
3. **启动时重置**：每次启动清空 Go 数据库，避免残留
4. **最终一致**：允许短暂不一致，通过启动恢复
