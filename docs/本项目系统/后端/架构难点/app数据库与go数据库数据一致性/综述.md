# App 数据库与 Go 数据库数据一致性

## 核心问题

本项目使用了一个**无法修改源码的 Go 后端**（`wx_video_download.exe`）来执行实际的视频下载。Go 后端自带 `gopeed.db` 数据库管理下载任务，但该数据库：

1. **无法扩展**：不能添加 `video_id`、`author_id` 等业务字段
2. **功能受限**：没有提供完整的任务管理 API
3. **数据不透明**：任务状态、进度格式多变（dict/int 混用）

因此，本项目**必须维护自己的 `app.db` 数据库**来存储业务数据（作者、视频、任务关联）。

**核心矛盾**：两个数据库各自独立，如何保证一致性？

---

## 实际策略：App 为主，Go 为辅

本项目采用**"App 数据库为主，Go 数据库为临时缓存"**的策略：

```

│  │  │             │  │ is_downloaded│  │ video_id关联   │  │    │

│                              │                                   │
```

### 关键设计决策

| 决策 | 原因 |
|------|------|
| **启动时清空 gopeed.db** | 避免 Go 数据库中的旧任务残留导致状态不一致 |
| **app.db 存储任务镜像** | 关联 `video_id`，支持按视频查询任务状态 |
| **WebSocket 监听进度** | 实时获取 Go 后端的下载进度，写入 app.db |
| **启动时校验文件** | 修复 "Go 报 done 但文件丢失" 的脏数据 |

## 为什么不能保证强一致性？

### 1. Go 后端不可控

```
Go 后端行为无法预测：
- 任务状态格式不统一（progress 有时是 dict，有时是 int）
- 错误信息不完整（error 字段可能为空）
- 文件存在性不可靠（报 done 但文件可能不存在）
```

### 2. 双写时序问题

```python
# 创建任务时必须先写 Go，再写 App
result = client.download_video(...)  # Go 创建任务
task_id = result["data"]["id"]

# 如果此时 App 崩溃，app.db 中没有记录
# 但 Go 数据库中有任务 → 状态不一致
db.create_download_task(task)  # App 创建镜像
```

### 3. WebSocket 断连风险

```
WebSocket 连接断开时：
- Go 后端进度变化无法推送到 App
- App 数据库中的进度过时
- 用户看到的状态与实际不符
```

## 实际采用的"妥协策略"

由于无法保证强一致性，本项目采用**"启动时重置 + 运行时同步 + 容错处理"**的策略：

### 策略 1：启动时清空 Go 数据库

```python
# core/api/app.py
def _auto_start_go_backend():
    # 启动前清空 gopeed.db，避免旧任务残留
    _clear_go_tasks_safe()  # 直接删除 gopeed.db 文件

    # Go 启动后清理 app.db 中的残留任务
    TaskService.cleanup_stale_tasks()
```

**效果**：每次启动时，Go 数据库是空的，App 数据库中的活跃任务会被清理，确保两边一致。

**代价**：重启后所有下载任务丢失，需要监控器重新创建。

### 策略 2：运行时单向同步（Go → App）

```python
# WebSocket 监听 Go 后端进度
def _on_message(self, ws, message):
    task = json.loads(message)["data"]["task"]

    # 终态时写 app.db
    if task["status"] in ("done", "completed"):
        db.update_video_downloaded(video_id, path)
        db.update_download_task_status(task_id, "done")
```

**效果**：Go 后端的进度变化实时同步到 App 数据库。

**局限**：WebSocket 断连时同步中断，需要轮询兜底。

### 策略 3：启动时校验文件完整性

```python
# core/service/task.py
def reconcile_orphaned_downloads():
    """启动时校验：is_downloaded=1 但文件不存在的记录重置为 0"""
    for video in db.list_all_videos():
        if video.is_downloaded == 1:
            if not os.path.isfile(video.download_path):
                db.reset_video_downloaded(video.video_id)
```

**效果**：修复 "Go 报 done 但文件丢失" 的脏数据。

### 策略 4：完成时验证文件存在

```python
# core/service/task.py
def get_task_progress(self, task_id):
    if status in ("done", "completed"):
        file_exists = os.path.isfile(full_path)
        if not file_exists:
            # Go 报 done 但文件不存在 → 标记为 failed
            db.update_download_task_status(task_id, "failed")
```

**效果**：防止 Go 后端误报完成状态。

## 一致性问题与解决方案汇总

| 问题 | 根因 | 解决方案 | 效果 |
|------|------|----------|------|
| Go 重启后任务丢失 | gopeed.db 被清空 | 启动时清理 app.db 残留任务 | 两边一致，但任务丢失 |
| 文件丢失但状态为已下载 | Go 报 done 但文件被删除 | 启动时校验文件完整性 | 自动重置脏数据 |
| Go 报 done 但文件不存在 | Go 后端误报 | 完成时验证文件存在 | 标记为 failed |
| WebSocket 断连 | 网络问题 | 自动重连 + 轮询兜底 | 最终一致 |
| 进度格式不统一 | Go 后端设计问题 | 标准化处理 | 兼容两种格式 |

## 为什么不使用 gopeed.db 的进度管理？

### 问题 1：无法关联业务数据

```
gopeed.db 任务表：
- id: 任务 ID
- name: 文件名
- status: 状态
- progress: 进度

缺失字段：
- video_id: 无法关联视频
- author_id: 无法关联作者
- video_type: 无法区分短视频/直播回放
```

### 问题 2：API 不完整

```
Go 后端提供的 API：
- POST /api/task/create2: 创建任务 ✓
- GET /api/task/list: 获取任务列表 ✓
- POST /api/task/delete: 删除任务 ✓

缺失的 API：
- 按视频 ID 查询任务状态
- 按作者 ID 查询任务列表
- 批量创建/删除任务
```

### 问题 3：数据格式不稳定

```javascript
// Go 后端 progress 字段有两种格式
// 运行中
{ "progress": { "downloaded": 1024, "speed": 512 } }

// 完成时
{ "progress": 100 }
```

## 最佳实践

1. **启动时清空 Go 数据库**：避免旧任务残留
2. **App 数据库存储任务镜像**：关联业务字段
3. **WebSocket + 轮询双保险**：确保进度同步
4. **终态必须验证文件**：不信任 Go 后端的 done 状态
5. **启动时校验文件完整性**：修复脏数据

## 新架构实现（2026-05-20）

### 已实现的改进

| 组件 | 文件 | 功能 |
|------|------|------|
| EventBus | `core/utils/event_bus.py` | Go 连接状态管理，事件发布订阅 |
| FileValidator | `core/utils/file_validator.py` | 启动时文件校验，任务前检查 |
| OrphanTaskCleaner | `core/utils/orphan_cleaner.py` | 启动时 + 定期后台清理孤立任务 |
| CompensationTransactionManager | `core/utils/compensation_manager.py` | 补偿事务，双写回滚 |
| ProgressSynchronizer | `core/utils/progress_synchronizer.py` | WebSocket + HTTP 轮询双保险 |

### 一致性保证

1. **双写操作**：补偿事务模式，App 失败自动回滚 Go
2. **监控器阻塞**：事件驱动，Go 离线自动暂停
3. **损坏文件**：启动时清理 + 任务前检查
4. **进度同步**：WebSocket 正常时实时，断开时轮询
5. **孤立任务**：启动时全量 + 定期（10分钟）增量清理

### 启动流程

```
应用启动
    │
    ├─► [1] 创建事件总线（最先）
    │       get_event_bus()
    │
    ├─► [2] 文件校验器 - 启动时校验
    │       FileValidator.validate_on_startup()
    │
    ├─► [3] 启动 Go 后端
    │       _auto_start_go_backend()
    │
    ├─► [4] 孤立任务清理器 - 启动时清理
    │       OrphanTaskCleaner.cleanup_on_startup()
    │
    ├─► [5] 启动 WebSocket 监听器（发布连接事件）
    │       DownloadProgressListener(event_bus=event_bus)
    │
    ├─► [6] 启动进度同步器
    │       ProgressSynchronizer(event_bus=event_bus)
    │
    └─► [7] 启动定期清理
            OrphanTaskCleaner.start_periodic_cleanup()
```
